# ADR-0002: 快照模式——排课验证报告的持久化方案

## 状态

已采纳

## 上下文

需求要求排课系统输出"验证报告"——包含硬约束合规状态和软约束评分明细。用户需对比多次排课/微调后结果。

候选方案：
- **实时模式**：不存报告，每次查看时实时计算。永远反映最新状态。
- **快照模式**：排课/微调后生成持久化快照，可追溯历史。

## 决策

**采用快照模式（A），辅以轻量实时预览（B）。**

快照有两张表：
- `ScheduleSnapshot`：主表，记录硬约束状态、四项软约束得分、综合分、统计信息
- `SnapshotDetail`：明细表，按教师×约束维度的得分贡献，支持"拖后腿排行榜"

触发方式：
- **自动**：排课引擎完成后自动生成快照（Trigger=auto）
- **手动**：微调后用户主动点击"生成报告"（Trigger=manual）

微调时提供轻量实时评分预览（当前分数变化），但不生成快照，避免高频操作产生大量中间记录。

## 理由

1. **可对比**：需求明确要求"支持重新生成以获取更高评分方案"——隐含历史对比需求。实时模式无法对比。

2. **可存证**：验证报告是"官方文档"属性，需生成时间戳，支持 PDF 导出归档。

3. **可控**：手动触发给用户控制权——微调频繁时不自动产生快照，满意后再出正式报告。

## 后果

- **正面**：历史对比、PDF 导出、评分趋势追踪均基于快照实现
- **负面**：快照占用存储空间（每条快照约 5-20KB，可忽略）
- **相关**：验证报告页（ReportPage）消费快照数据；ScoringService 同时服务于排课求解器（实时评分）和快照生成（持久化评分）
