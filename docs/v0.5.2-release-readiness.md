## 5. v0.5.3 详细设计建议 — 课程类别 × 教室类型 × 设备约束

### 5.1 问题分析

当前系统对教室资源的约束仅为 **硬约束**：
- `Classroom.Capacity >= ClassGroup.Students`（容量）
- 时间冲突检查（同时段一教室一课）

缺失的约束维度：
1. **课程类别 vs 教室类型**：实验课需要实验室，体育课需要体育馆，机房课需要机房
2. **设备约束**：某些课程需要投影仪、智慧黑板、空调等特定设备
3. **软偏好**：理论课优先排普通教室，把实验室留给实验课

### 5.2 设计原则

**最小侵入 Stable Core**：
- 不修改 `ScoreSchedule` 签名
- 不修改 7 个软约束公式
- 不修改 `ScoringContext` 核心字段
- 全部以模型字段追加 + 求解器约束增强 + 可选软约束扩展

### 5.3 模型变更（append-only）

```go
// Course 新增
type Course struct {
    // ... existing fields ...
    Category string `gorm:"size:20;default:theory" json:"category"` 
    // theory | lab | pe | seminar | computer
}

// Classroom 新增
type Classroom struct {
    // ... existing fields ...
    RoomType  string `gorm:"size:20;default:standard" json:"roomType"`
    // standard | lab | computer | gymnasium | multimedia
    
    Equipment string `gorm:"size:500" json:"equipment"` 
    // JSON array: ["projector","smartboard","aircon","lab_bench","computers"]
}

// TeachingTask 新增
type TeachingTask struct {
    // ... existing fields ...
    RequiredRoomType  string `gorm:"size:20" json:"requiredRoomType,omitempty"`
    // 空 = 不指定, 由 Course.Category 推导
    
    RequiredEquipment string `gorm:"size:500" json:"requiredEquipment,omitempty"`
    // JSON array, 空 = 无设备要求
}
```

### 5.4 约束匹配规则

**硬约束（求解器内）**:

```
matchRoom(task, room) =
    (task.RequiredRoomType == "" || room.RoomType == task.RequiredRoomType)
    && (task.Course.Category == "pe"     → room.RoomType == "gymnasium")
    && (task.Course.Category == "lab"    → room.RoomType == "lab")
    && (task.Course.Category == "computer" → room.RoomType == "computer")
    && hasAllEquipment(room.Equipment, task.RequiredEquipment)
```

**推导规则**（当 `RequiredRoomType` 为空时）:

| Course.Category | 推导 RoomType |
|-----------------|---------------|
| theory | standard |
| lab | lab |
| pe | gymnasium |
| computer | computer |
| seminar | standard \| multimedia |

### 5.5 评分影响（可选软约束）

**方案**: 新增第 8 个软约束 `roomTypePreference`（可选，默认不启用）

```go
// scoreRoomTypePreference: 同类课程优先集中教室
// 
// 启用条件: ScoringContext.EnabledConstraints 包含 "room_type_preference"
// 评分逻辑: 统计每个 (Course.Category, RoomType) 组合的分布，
//   奖励 "理论课→普通教室" 的比例，惩罚 "理论课→实验室" 的占用。
// 满分: 100 (与其他维度一致)
```

**Stable Core 影响**: 
- `ScoringContext.EnabledConstraints` 已存在，只需在 `ScoreSchedule` 内新增 `if contains("room_type_preference")` 分支
- `ScoreBreakdown` 追加 `RoomTypePreference float64` 字段
- `PerCategoryMax` 和 `EnabledCategoryCount` 自动适配
- **不修改** 7 个已有软约束的任何代码

### 5.6 求解器变更

**SA 求解器** (`sa_initial.go` / `sa_neighbors.go`):
- `buildInitial()`: 教室候选列表增加 `matchRoom()` 过滤
- `tryMove()` / `trySwap()`: 目标教室需通过 `matchRoom()` 检查
- **增量影响**: 仅在候选生成阶段过滤，不改变邻域搜索结构

**OR-Tools** (`solver.py`):
- 新增约束: `model.Add(room_type[task] == required_type[task])`
- 设备约束: `model.AddAllDifferent` 或布尔变量组合

### 5.7 数据迁移

```sql
-- 无需迁移。新字段全部 default 空值/standard，旧数据自动兼容。
-- Course.Category 默认 "theory" → 推导 RoomType "standard" → 全部教室可用
-- 等同于 v0.5.2 行为。
```

### 5.8 前端变更

- 课程管理: 新增 Category 下拉选择

- 教学任务: 新增 RequiredRoomType (可选) + RequiredEquipment (可选)
- 排课页: 无变更（约束在求解器内自动应用）
