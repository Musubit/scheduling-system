## 一、Context（本次修订原因）

用户在 Stage B 决策确认中给出三条约束：

1. **Building**：Code 与 Name 分离（Code 用于解析/关联，Name 用于 UI 展示）
2. **Seed**：只提供代表性教学资源，覆盖 6 种 RoomType，**不追求完整录入**
3. **RoomCodeParser**：支持 `1-302 / 1-001 / 6A-422 / 7B-301`，非法输入返回 warning，**禁止硬阻断导入**

**核心禁令**（用户原文）：
- ❌ 将湖工大特殊规则写死进核心模型
- ❌ Building 使用 A/B/C 抽象编号
- ❌ Seed 承担真实学校完整数据职责

**架构目标**（用户新增）：

> 建立**通用 Resource Domain**，为后续 **TimeScheduler / RoomScheduler 两阶段架构**提供稳定资源层。

这个目标决定了 Building/Classroom 必须是**学校无关**的抽象，parser 必须完全隔离，seed 数据只是演示性。

## 二、Design Principles（本次核心约束）

| 层次 | 责任 | 是否含 HBUT 特殊规则 |
|---|---|---|
| **Core Domain**（`models/building.go` / `classroom.go` / `room_types.go`） | 通用资源抽象：Code + Name + Floor + Number + RoomType 枚举 | ❌ **不含** |
| **Adapter Layer**（`services/parsers/hbut/`） | HBUT 特定的编号解析规则（6教=LAB / 分A/B/C 区） | ✅ 集中于此 |
| **Seed Data**（`database/seed.go`） | 演示性代表数据，10-15 间覆盖 6 种 RoomType + 分区楼演示 | 🟡 使用 HBUT 编号但数量最小 |
| **Solver / ResourceMatcher** | 只消费 Core Domain，通过 RoomType 枚举匹配 | ❌ **不含** |
| **Frontend UI** | Building 下拉 / RoomType i18n；不解析编号 | ❌ **不含** |

**关键设计**：Core 中 `Classroom.Code` 只是**唯一标识字符串**，无内在结构约束。Parser 是**外部工具**，Core 不知其存在。任何学校（湖工大 / 华科 / 武大）都能用同一 Core，只需换 adapter。

## 三、Domain Model Design

### 3.1 Building 模型（通用）

```
type Building struct {
    gorm.Model
    Code     string `gorm:"uniqueIndex;size:20;not null" json:"code"`  // 学校自定义，如 "1"/"5A"/"6A"/"MB1"
    Name     string `gorm:"size:100;not null" json:"name"`              // UI 展示名，如 "1 教"/"5 教 A 区"
    Category string `gorm:"size:20;default:teaching" json:"category"`   // teaching / lab / sports / other
    Status   string `gorm:"size:20;default:active" json:"status"`
}
```

**Category 常量**（通用，非 HBUT 特定）：

```
const (
    BuildingCategoryTeaching = "teaching"  // 教学楼（含普通教室 + 多媒体 + 阶梯）
    BuildingCategoryLab      = "lab"       // 实验楼
    BuildingCategorySports   = "sports"    // 体育场馆
    BuildingCategoryOther    = "other"     // 其他
)
```

**Category 语义**：仅描述楼栋的**整体功能定位**，不硬约束该楼内的教室类型（例：教学楼里也可以有机房；实验楼里也可以有多媒体教室 —— 这是 seed / 用户配置决定，不是模型约束）。

### 3.2 Classroom 模型升级（通用）

```
type Classroom struct {
    gorm.Model
    Code       string `gorm:"uniqueIndex;size:20;not null" json:"code"`   // 学校自定义唯一编号
    Name       string `gorm:"size:100;not null" json:"name"`               // UI 展示名（默认同 Code）
    BuildingID uint   `gorm:"index;not null" json:"buildingId"`             // FK → buildings.id
    Floor      int    `gorm:"not null;default:1" json:"floor"`              // 楼层
    Number     string `gorm:"size:10" json:"number"`                        // 教室号（不含楼栋前缀），可空
    Capacity   int    `gorm:"not null;default:0" json:"capacity"`
    RoomType   string `gorm:"size:20;not null;index" json:"roomType"`      // 英文枚举，Required
    Status     string `gorm:"size:20;default:available" json:"status"`
    Equipment  string `gorm:"type:text" json:"equipment"`                   // v0.5.3 JSON array

    Building Building `gorm:"foreignKey:BuildingID" json:"building,omitempty"`
}
```

**字段变更清单**：

| 变更 | 字段 | 说明 |
|---|---|---|
| 🔴 删除 | `Building string` | 冗余字符串，改为 FK |
| 🟢 新增 | `BuildingID uint` FK | 关联 Building 表 |
| 🟢 新增 | `Number string` | 教室号（不含楼栋前缀），可空 |
| 🔴 重命名 | `Type` → `RoomType` | 语义更明确 |
| 🔴 值域重写 | 中文 → 英文枚举 | 见 §3.3 |
| ✅ 保留 | `Code / Name / Floor / Capacity / Status / Equipment` | 保持 |

**关键设计**：Core 不定义"Code 必须是 `A-BCD` 格式" —— Code 是**学校自定义字符串**。所有 Code 解析工作在 adapter 层。

### 3.4 RoomCodeParser 定位（Adapter 层，完全隔离于 Core）

**目录结构**：

```
backend/
├── models/                    # Core Domain（通用）
│   ├── building.go
│   ├── classroom.go
│   └── room_types.go
├── services/
│   ├── parsers/               # Adapter 层 — 学校特定规则
│   │   └── hbut/
│   │       ├── room_code.go       # HBUT 编号解析
│   │       └── room_code_test.go
│   ├── resource_matcher.go    # 只消费 Core（不引用 parsers/）
│   └── ...
```

**关键约束**：
- `backend/models/**` **禁止** import `backend/services/parsers/**`
- `backend/services/resource_matcher.go` / `sa_solver.go` / `move_service.go` **禁止** import `backend/services/parsers/**`
- 只有 **UI 导入路径**（未来可能引入）和 **seed / test fixtures** 才能引用 parser

**HBUT Parser 接口**：

```
package hbut

// ParsedRoomCode 是 HBUT 编号解析的结果，仅在 adapter 层内部使用。
type ParsedRoomCode struct {
    BuildingCode  string   // "1", "5A", "6A", "7B"
    Floor         int      // 楼层
    RoomNumber    string   // 教室号（不含楼栋前缀）
    RoomTypeHint  string   // "LECTURE"（0xx）/"LAB"（6教）/""
    Warnings      []string // 解析警告（非法字符、格式异常等），非致命
}

// Parse 尝试按 HBUT 编号规则解析。
// 非法输入时：
//   - 尽量提取可识别部分
//   - Warnings 记录问题
//   - **不返回 error 硬失败**（禁止阻断导入流程）
//
// 空字符串输入返回全零值 + Warnings("empty code")。
func Parse(code string) ParsedRoomCode
```

**解析规则**（仅在 hbut adapter 内实现）：

- `-` 分隔 BuildingCode 与 RoomNumber
- `-` 前部分作为 BuildingCode（可含字母，如 `5A`, `7B`）
- `-` 后第一位为 Floor（0 = 阶梯大教室提示）
- `-` 后剩余为 RoomNumber
- BuildingCode 首字符为 `"6"` → RoomTypeHint = "LAB"
- Floor == 0 → RoomTypeHint = "LECTURE"（优先级高于 6 教规则）
- 无法解析的字段留空 + Warnings

**示例**：

| Input | BuildingCode | Floor | RoomNumber | RoomTypeHint | Warnings |
|---|---|---|---|---|---|
| `1-302` | 1 | 3 | 02 | "" | — |
| `1-001` | 1 | 0 | 01 | LECTURE | — |
| `6A-422` | 6A | 4 | 22 | LAB | — |
| `7B-301` | 7B | 3 | 01 | "" | — |
| `foo-bar` | foo | 0 | bar | "" | ["floor part 'bar' non-numeric"] |
| ``（空） | "" | 0 | "" | "" | ["empty code"] |
| `1-` | 1 | 0 | "" | LECTURE | ["missing number part"] |

**用途**：
- 未来 Excel 批量导入：Parser 提取建议字段，UI 让用户确认后保存
- **不用于**排课/评分/匹配（这些逻辑只依赖 Core 的 RoomType 枚举）

### 3.5 与 Solver 的边界

**Solver 只依赖 Core**：
- `resource_matcher.go`：通过 `Classroom.RoomType` 枚举匹配
- `sa_solver.go`：过滤体育课 → `classroom.RoomType == models.RoomTypeGym`
- `ortools_client.go`：payload 中传 `RoomType string`，Python 侧只判等

**Solver 不知**：教室编号规则、楼栋是否分 A/B、6 教是否实验楼

这保证了未来 TimeScheduler / RoomScheduler 两阶段架构可以基于 Core 拆分，不受 adapter 层变化影响。

## 四、Seed Design（代表性 10-15 间）

### 4.1 Buildings Seed（5 座代表性）

覆盖各 Category + 分区演示：

| Code | Name | Category | 说明 |
|---|---|---|---|
| 1 | 1 教 | teaching | 单区教学楼代表 |
| 5A | 5 教 A 区 | teaching | 分区楼 A 侧 |
| 5B | 5 教 B 区 | teaching | 分区楼 B 侧 |
| 6A | 6 教实验楼 A 区 | lab | 实验楼代表 |
| GYM | 体育馆 | sports | 体育场馆代表 |

**共 5 座**。规模刚好演示所有 Building.Category + 分区能力。**不追求完整**。

### 4.2 Classrooms Seed（12 间代表性）

覆盖 6 种 RoomType + 楼栋分布：

| Code | Building | Floor | Number | Capacity | RoomType | 说明 |
|---|---|---|---|---|---|---|
| 1-001 | 1 | 0 | 001 | 220 | LECTURE | 阶梯 |
| 1-302 | 1 | 3 | 02 | 80 | NORMAL | 普通教室 |
| 1-501 | 1 | 5 | 01 | 100 | MULTIMEDIA | 多媒体 |
| 1-503 | 1 | 5 | 03 | 50 | COMPUTER | 机房 |
| 5A-303 | 5A | 3 | 03 | 70 | NORMAL | 分区楼 A |
| 5A-402 | 5A | 4 | 02 | 100 | MULTIMEDIA | 分区楼 A 多媒体 |
| 5B-201 | 5B | 2 | 01 | 90 | NORMAL | 分区楼 B |
| 5B-001 | 5B | 0 | 01 | 250 | LECTURE | 分区楼 B 阶梯 |
| 6A-213 | 6A | 2 | 13 | 40 | LAB | 实验楼 |
| 6A-422 | 6A | 4 | 22 | 40 | LAB | 实验楼 |
| 6A-105 | 6A | 1 | 05 | 50 | LAB | 实验楼 |
| GYM-01 | GYM | 1 | 01 | 300 | GYM | 体育馆 |

**共 12 间**。RoomType 分布：NORMAL × 3 / MULTIMEDIA × 2 / LECTURE × 2 / LAB × 3 / COMPUTER × 1 / GYM × 1 = 12。

**Stage A 兼容性**：Stage A 已在 seed 中添加 F201 机房，Stage B **完全替换**为上述 12 间真实编号（F201 消失，由 `1-503` 承担机房角色）。

**Rationale**：
- 12 间足够 Stage A 的 12 个 TeachingTask 排课不冲突（每 TT 1 间即可，各 RoomType 覆盖）
- 分区楼（5A/5B）演示 Building 分区能力
- 实验楼（6A）演示 Building.Category="lab" 语义
- 阶梯教室（`x-001` 格式）演示 Parser 的 LECTURE hint
- 无 A栋/B栋 抽象命名

### 4.3 Seed 不做 Parser 依赖

Seed 直接构造 Classroom 记录（`{Code, BuildingID, Floor, Number, RoomType}` 全部**明文指定**），**不调用 Parser**。理由：
- 保持 seed → Core 单向依赖
- Parser 是导入辅助工具，seed 是明确的固定数据
- 未来测试可用 Parser 单测覆盖同样的编号，交叉验证

## 六、Migration Strategy

### 6.1 无生产数据 → DROP + 重建

在 `database.go` InitDB 中追加一次性迁移函数（对齐现有 `MigrateDepartmentsFromSettings` 模式）：

```
// MigrateRoomDomainV055 一次性迁移：DROP 旧 classrooms 表（若存在旧字段结构）。
// 幂等：通过 setting key "room_domain_v055_migrated" 标记，重复调用无害。
func MigrateRoomDomainV055(db DB) {
    if isMarked(db, "room_domain_v055_migrated") {
        return
    }
    _ = db.Migrator().DropTable(&models.Classroom{})
    markSetting(db, "room_domain_v055_migrated", "true")
}
```

调用点：在 `AutoMigrate` 之前执行（确保 AutoMigrate 用新结构建表）。

**幂等性**：通过 `settings` 表的 flag key 保证 —— 首次执行 DROP + mark，后续启动看到 mark 即跳过。

### 6.2 Bindings 重生成

- `wails3 generate bindings -ts` 一次性

### 6.3 回滚

- 无生产数据，回滚 = revert commits 即可
- 无其他表 FK 依赖 buildings，回滚不影响其他表

## 十、Future Architecture Alignment（TimeScheduler / RoomScheduler 两阶段）

本 Stage B 为未来两阶段架构提供稳定资源层：

```
┌─────────────────────────────────────────────────────────┐
│  TimeScheduler（Phase A）— 时间维度决定                    │
│    输入：TeachingTask + Semester + AcademicTerm            │
│    输出：(TeachingTask, DayOfWeek, StartPeriod, Weeks)    │
│    不消费：Classroom / Building                             │
└─────────────────────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────────────────────┐
│  RoomScheduler（Phase B）— 空间维度决定                    │
│    输入：TimeScheduler 输出 + Classroom + Building         │
│    输出：ClassroomID                                       │
│    依赖：ResourceMatcher（本 Stage B 提供的 RoomType 枚举） │
└─────────────────────────────────────────────────────────┘
```

本 Stage B 保证：
- Classroom.RoomType 是稳定枚举，RoomScheduler 可直接消费
- Building.Category 是稳定枚举，RoomScheduler 未来可按楼栋偏好分组
- Parser 隔离在 adapter 层，学校换了 RoomScheduler 也不动
- Frontend 通过 i18n map 展示中文，多校部署换 map 即可

## 十二、Design Rationale Summary（回应用户三条约束）

| 用户约束 | 本设计如何满足 |
|---|---|
| Building Code 与 Name 分离 | `Building.Code` 唯一索引供关联，`Building.Name` 供 UI 展示，2 字段严格分工 |
| Seed 仅提供代表性资源 | 5 座 Buildings + 12 间 Classrooms，覆盖 6 种 RoomType + Building 分区演示；不追求完整 |
| Parser 支持指定格式 + warning 非阻断 | `hbut.Parse` 返回 `ParsedRoomCode{..., Warnings []string}`；无 error 返回值；非法输入尽量提取部分字段 |
| 禁止 HBUT 规则写死进核心 | Core `models/` 无任何 HBUT 特定字段/常量；Parser 独立在 `services/parsers/hbut/`；Core 不 import parser |
| 禁止 Building 使用 A/B/C 抽象编号 | Seed 使用真实编号 `1/5A/5B/6A/GYM`；单元测试若需 fixture 也用真实格式 |
| Seed 不承担完整数据职责 | 12 间样本教室，不覆盖所有真实教学楼；用户日后通过 UI 或导入功能补齐 |
