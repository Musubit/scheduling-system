# ADR-0005: SA + OR-Tools 多引擎架构与 uv 打包

## 状态

已采纳

## 上下文

ADR-0001 否决了 OR-Tools 作为主要求解器，理由包括体积激增 150MB+、MSVC/MinGW ABI 不兼容、Go 绑定不成熟。但 OR-Tools CP-SAT 确实提供了 SA 无法实现的能力：可证明的最优性或近似最优解，以及更丰富的约束建模（如教室容量约束传播）。

与此同时，用户反馈对 OR-Tools 的精确求解仍有需求，但不想增加部署复杂度。需要在"轻量单文件部署"和"精确求解能力"之间找到平衡。

## 决策

**采用多引擎架构：SA 优化版为主引擎，OR-Tools 为可选增强引擎，uv 管理 Python 依赖实现自包含打包。**

### 架构

```
┌──────────────────────────────────────────────┐
│  SchedulingService                          │
│                                            │
│  SolverOrchestrator (引擎编排)              │
│  ┌──────────────────────────────────────┐  │
│  │ 1. 尝试 OR-Tools（若用户选择）        │  │
│  │ 2. 不可用/超时 → 自动降级到 SA       │  │
│  │ 3. 用户可手动指定引擎（高级选项）     │  │
│  └──────────────────────────────────────┘  │
│                                            │
│  SA Solver (C+D)         ORToolsClient     │
│  - 多轮重启(3轮)          - HTTP 通信       │
│  - 贪心后优化              - localhost:19877│
│  - 纯Go，零依赖           - 超时/失败降级   │
└────────────────────────────────────────────┘
         │                      │
         │              ┌───────▼────────┐
         │              │ Python Flask   │
         │              │ solver.py      │
         │              │ (uv .venv)     │
         │              └────────────────┘
```

### SA 引擎增强（C + D）

- **C - 多轮重启**：默认跑 3 轮 SA，使用不同随机种子，时间预算均分，取最高分结果。首轮贪心初始解，后续轮随机初始解增加多样性。
- **D - 贪心后优化**：SA 最优解产出后，识别得分最低的 N 个条目（默认 top 10%），对每个条目暴力搜索所有可行的（时段 × 教室）组合，接受任何提升总分的替换。

### OR-Tools 引擎（可选）

- Python Flask 微服务，`POST /solve` 接收排课数据，CP-SAT 建模求解
- Go 侧通过 `os/exec` 启动 Python 进程，HTTP 通信
- 自动降级：OR-Tools 不可用/超时 → 无缝切换到 SA，用户无感知

### uv 打包方案

- 使用 `uv` 创建自包含虚拟环境（`.venv/`），包含 Python + ortools + flask
- 整个 `scheduler/` 目录嵌入 Wails 资源
- 应用启动时自动解压并拉起 Python 服务
- 对比 PyInstaller：体积更可控，不重复打包 Python 解释器

## 理由

1. **SA 保持为零依赖主引擎**：ADR-0001 的轻量优势不变，新增的多轮重启和后优化显著提升解质量
2. **OR-Tools 作为可选增强**：有 Python 环境的用户可以享受精确求解，没有的用户仍能正常使用 SA
3. **uv 解决 Python 依赖管理**：自包含虚拟环境，用户不需要手动安装 Python 或 pip
4. **自动降级保证可用性**：默认"智能模式"，用户只需点一个按钮，引擎选择对用户透明
5. **渐进式采用**：SA 的改进（C+D）立即可用，OR-Tools 可在后续迭代中逐步完善

## 后果

- **正面**：满足不同用户需求（轻量 vs 精确），SA 结果质量显著提升，架构可扩展支持未来更多引擎
- **负面**：维护两套求解路径增加了代码复杂度；uv 打包增加约 50-100MB 体积（可选，用户可删除 scheduler 目录使用纯 SA 模式）
- **可逆性**：SolverOrchestrator 的降级逻辑确保即使 OR-Tools 完全移除，SA 仍可独立工作
