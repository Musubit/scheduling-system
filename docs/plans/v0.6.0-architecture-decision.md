## 1. Executive Summary

v0.6.0 的核心目标是 **完成调度内核迁移**——消除当前 TIME_ONLY/FULL 双模式架构中的耦合问题，将调度编排从巨石函数 `SchedulingService.RunScheduling()`（600+ 行）迁移到新的 `scheduling/` 分层 Pipeline，并冻结核心领域模型。

**三个核心变化**:
1. **ScheduleEntry 拆分为 TimeAssignment + ScheduleEntry** — 时间排课与教室分配解耦
2. **SchedulingOrchestrator 接管生产路径** — 以 Adapter 模式渐进替换
3. **TimeScheduler 延后至 v0.6.1 实现** — v0.6.0 先完成架构层（模型拆分 + Pipeline 接管）

**不在此范围**:
- 新评分维度（v0.7）
- AI 辅助（v0.8）
- Installer / 自动更新（v0.9）

## 4. Decision Records

### Decision 1: ScheduleEntry 拆分策略

#### Context

当前 `ScheduleEntry` 同时承载时间信息（DayOfWeek, StartPeriod, Span, TeacherID, CourseID）和资源信息（ClassroomID）。TIME_ONLY 模式下通过 `buildVirtualClassroomsForTimeOnly()` 创建虚拟教室 + `applySyntheticClassroomIDsForTimeOnly()` 标记 `IsVirtual=true` 来绕过——这违反了 Spec §2.3 INV-E1（TIME_ONLY 下 schedule_entries 零行）。

#### Options

**方案 A: ClassroomID 改为 nullable (`*uint`)**

改动范围:
- `ScheduleEntry.ClassroomID` 类型变更: `uint` → `*uint`
- `idx_schedule_room` 唯一索引移除 ClassroomID 列（或改为 partial unique index）
- TIME_ONLY 下 ClassroomID 写 NULL
- 评分 `scoreClassrooms = nil` 跳过资源维度（已有）
- 删除 `buildVirtualClassroomsForTimeOnly()` 和 `applySyntheticClassroomIDsForTimeOnly()`

工作量估算: ~4h (模型变更 + 索引调整 + 删除 hack)

**方案 B: 拆分为 TimeAssignment + ScheduleEntry**

改动范围:
- TimeAssignment 模型已存在（`backend/models/time_assignment.go`），通过 AutoMigrate 已建表
- ScheduleEntry 重构为最小模型（仅 TimeAssignmentID + ClassroomID）
- 写路径: `SchedulingService` 在事务中先写 TimeAssignment，再写 ScheduleEntry（仅 FULL）
- 读路径: 需新增 `ScheduleQueryService.GetEnrichedScheduleEntries()`，JOIN 三表（ta + se + tt + teacher + course + classroom）返回前端消费的 `EnrichedScheduleEntry`
- MoveService: `CheckMove` / `MoveEntry` / `MoveEntryAndScore` 需适配新模型
- Snapshot/Version: JSON 内容从旧 `[]ScheduleEntry` 迁移到 `ScheduleSnapshotDTO`
- `solver.py`: 需新增 `solve_time_only` 路径（或让 Go 侧 Adapter 过滤）
- 前端: Wails bindings 重新生成，UI 组件通过 `EnrichedScheduleEntry` 消费（透明切换）

工作量估算: ~20h (模型 + 迁移 + 写路径 + 读路径 + MoveService + snapshot + 前端适配)

#### Analysis

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| INV-E1 合规 | ⚠️ 部分: TIME_ONLY 下 ClassroomID = nil，但仍有行 | ✅ 完全: TIME_ONLY 下 schedule_entries 零行 |
| 架构纯粹性 | ❌ 时间+教室仍耦合在同一模型 | ✅ 时间与教室彻底解耦 |
| v0.7 多方案排课 | ❌ 需额外改造 | ✅ 自然支持: 同一 TimeAssignment 可对应多个 ScheduleEntry（不同方案） |
| v0.8 AI 辅助调整 | ❌ 改教室=改 ScheduleEntry，改时间也=改 ScheduleEntry | ✅ 改时间→TA，改教室→SE，操作清晰 |
| 前端影响 | 🟢 仅移除 IsVirtual 字段 | 🟡 EnrichedScheduleEntry 适配 |
| MoveService 影响 | 🟢 几乎无变化 | 🟡 需适配新模型读写 |
| 数据迁移风险 | 🟢 仅修改列约束 | 🔴 表结构变更，需迁移脚本 |
| 工作量 | ~4h | ~20h |

#### Recommendation

**方案 B** — 拆分为 TimeAssignment + ScheduleEntry。

#### Reason

1. **设计一致性**: Spec (`2026-07-13-scheduling-dual-mode-design.md`) 已经冻结了 TimeAssignment + ScheduleEntry 的目标架构，方案 A 是临时妥协，方案 B 是 spec 的原生设计。
2. **时间与教室彻底解耦**: 方案 A 只是让 `ClassroomID` 可以为空，时间数据和教室数据仍在同一张表——未来任何需要"只看时间"或"只看教室"的操作都要在代码中过滤，而不是在数据模型中表达。
3. **v0.7 Multi-Solution 的天然基础**: 多方案排课（教师满意最高 vs 教室利用最高 vs 学生体验最高）的关键是 **同一组时间分配对应不同教室分配**。方案 B 的 1:1 结构天然支持，方案 A 需要复制整行。
4. **v0.8 AI 辅助调整**: AI 建议"改时间"和"改教室"是两个独立操作。方案 B 中改时间 = 修改 TimeAssignment，改教室 = 修改 ScheduleEntry。方案 A 中两者都在同一个对象上。
5. **TimeAssignment 模型已落地**: 表已通过 AutoMigrate 创建，索引已就绪，schema_migrations 已支持——不是从零开始。
6. **设计原则**: "先架构稳定，再算法增强"。方案 B 先稳定数据模型，v0.7 再增强算法——不需要在 v0.6 和 v0.7 之间做两次数据迁移。

### Decision 2: Pipeline 迁移策略

#### Context

当前生产路径直接调用 SA solver 和 OR-Tools client，Orchestrator 概念只存在于 `scheduling/orchestrator/` 包中（108 行，未被生产路径引用）。

#### Options

**方案 A: 一次性重写 (Big Bang)**

在 v0.6.0 中完成:
1. TimeAssignment + ScheduleEntry 模型切换
2. 旧 SA solver 提取为 `scheduling/time/` 纯 TimeScheduler
3. 旧评分服务重写为 `scheduling/score/` 4-bucket Scorer
4. 旧资源匹配器替换为 `scheduling/matcher/`
5. `SchedulingOrchestrator` 接管 `SchedulingService`
6. `solver.py` 改造为 time-only
7. 删除所有旧代码

**方案 B: 渐进迁移 (Strangler Fig + Adapter)**

分步推进:
1. v0.6.0: 模型切换（TimeAssignment + ScheduleEntry）+ Orchestrator Adapter 接管
2. v0.6.1: Solver 迁移到 `scheduling/time/` + 旧代码删除
3. v0.6.2: 收尾清理 + 模型冻结

Adapter 模式:
```
SchedulingOrchestrator (新)
    │
    ├── ITimeScheduler = LegacySASolverAdapter
    │   └── 内部调用: NewSASolver().SolveMultiRun(...)
    │   └── 转换: ScheduleEntry → TimeAssignmentDraft
    │
    ├── IRoomScheduler = LegacyRoomAllocatorAdapter
    │   └── 内部: 从 ScheduleEntry 提取教室分配
    │   └── (v0.6.1 替换为 Greedy RoomScheduler)
    │
    └── IScorer = NewScorerAdapter
        └── 内部调用: NewScoringService().ScoreSchedule(...)
        └── (v0.6.1 替换为 4-bucket Scorer)
```

#### Analysis

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 风险 | 🔴 极高: 一次改动 ~3000 行 | 🟡 可控: 每次改动 ~500 行 |
| 回滚能力 | 🔴 困难: 所有代码交织 | 🟢 简单: revert 单个 commit |
| 测试粒度 | 🔴 需全量回归 | 🟢 可逐阶段验证 |
| 开发周期 | ~22 工作日 (整块) | ~22 工作日 (分 3 个小版本) |
| 中间状态 | 无: 全部完成才能 merge | 每个版本可独立 merge |
| Adapter 浪费 | 无 | 临时 Adapter (~200 行，v0.6.2 删除) |

#### Recommendation

**方案 B** — 渐进迁移 (Strangler Fig + Adapter)。

#### Reason

1. **与原始 Spec 一致**: Spec §5.1 明确选择了 Strangler Fig 策略，三条纪律:
   - 旧代码在切换前不修改
   - 新代码在切换前不接入生产路径
   - 切换是原子、可逆的 PR
2. **v0.6 分三个子版本正好匹配**: v0.6.0 模型 + 编排接管，v0.6.1 Solver 迁移，v0.6.2 清理冻结。
3. **Adapter ~200 行是已知的临时成本**: 代码量小，删除时风险极低。
4. **生产排课不能中断**: 渐进迁移允许每个版本都经过完整回归，不会出现"改到一半排课功能不可用"的中间状态。
5. **原始 Spec 的 12-PR 计划**: 方案 B 与该计划完全对齐（PR-09 模型切换 → PR-10 入口切换 → PR-11 清理）。

### Decision 3: TimeScheduler 实现时机

#### Context

`scheduling/time/` 目录不存在。`ITimeScheduler` 接口已定义。旧 SA solver (`sa_solver.go`, 517 行) 混合了时间排课和教室分配逻辑——`MoveRoom`、`roomOcc`、`AllowedRooms` 等。

SA solver 的教室耦合点:
- **sa_neighbors.go**: `MoveRoom` 操作（更换教室）+ `Swap` 含教室交换
- **sa_initial.go**: 贪心初始解同时分配时间和教室
- **sa_postopt.go**: 后优化涉及教室再分配
- **sa_scorecache.go**: `roomOcc` (教室占用) 维度 + `lowFloorScore` (资源维度评分)
- **sa_solver.go**: `TeachingTaskData.AllowedRooms` + `RequiredRoomType`

#### Options

**方案 A: v0.6.0 直接拆出纯 TimeScheduler**

在 v0.6.0 同时完成:
1. 从旧 SA solver 提取时间排课逻辑
2. 剥离所有教室相关代码
3. 创建 `scheduling/time/` 实现 `ITimeScheduler`
4. SA score cache 从 4 维度降为 3 维度 (Time/Teacher/Student)
5. OR-Tools solver.py 改造为 time-only 或创建新路径

**方案 B: v0.6.0 Adapter 包装旧 Solver，v0.6.1 再拆**

v0.6.0:
- 创建 `LegacySASolverAdapter` 实现 `ITimeScheduler` 接口
- Adapter 内部调用旧 `NewSASolver().SolveMultiRun()`
- Adapter 负责: ScheduleEntry ↔ TimeAssignmentDraft 转换 + 评分桥接
- 目标: Orchestrator 跑通，但底层 solver 不变

v0.6.1:
- 创建 `scheduling/time/` 纯 TimeScheduler
- 从旧 SA 提取时间逻辑
- 替换 Adapter
- 删除旧 solver 代码

#### Analysis

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| v0.6.0 工作量 | 🔴 ~30h (模型 + Solver + 编排) | 🟡 ~20h (模型 + 编排 Adapter) |
| SA 提取复杂度 | 🔴 高: SA 时间/教室逻辑交织 | 🟢 延后处理，责任分离 |
| v0.6.0 风险 | 🔴 模型和算法同时变 | 🟡 仅模型变，算法不变 |
| v0.6.1 自由度 | N/A | 🟢 专注 Solver，无模型干扰 |
| Score cache 改造 | 🔴 4维→3维 需同时完成 | 🟢 v0.6.1 再做 |
| Python solver.py | 🔴 需同步改造 | 🟢 Adapter 层转换 |

#### Recommendation

**方案 B** — v0.6.0 Adapter 包装，v0.6.1 拆 Solver。

#### Reason

1. **v0.6.0 已承担模型切换风险**: Decision 1 选了方案 B（模型拆分），这已经涉及数据库迁移、写路径重构、读路径适配、前端适配——不应同时叠加 Solver 重构。
2. **SA Solver 教室耦合深**: 517 行 SA 代码中，`MoveRoom`、`roomOcc`、`AllowedRooms`、`RequiredRoomType` 与时间排课逻辑深层交织。强行在同一个版本中拆分会引入不必要的回归风险。
3. **ROADMAP 明确分工**: v0.6 是"基础架构稳定化"，v0.7 才是"算法增强"。Solver 拆分属于架构层面（v0.6.1），Solver 优化属于算法层面（v0.7.0）。
4. **"先架构稳定，再算法增强"**: v0.6.0 稳定数据模型 → v0.6.1 迁移 Solver（架构收尾）→ v0.7.0 优化 Solver（算法增强）。
5. **Adapter 代码量可控**: `LegacySASolverAdapter` 约 150 行，只做数据转换和接口适配——代价低，收益高。

### 6.2 Migration 策略

**策略**: 硬切换（DROP + CREATE），因为:
- v0.5.x 无生产数据（本地 SQLite，用户可删除重建）
- ScheduleEntry 结构彻底变化，数据迁移不可行
- `time_assignments` 表已创建但无数据

**迁移步骤** (在 v0.6.0 PR-09 执行):

```sql
-- 单事务执行
BEGIN TRANSACTION;

-- 1. 备份旧 schedule_entries（如果存在数据）
CREATE TABLE IF NOT EXISTS _bak_schedule_entries_v054 AS
SELECT * FROM schedule_entries;

-- 2. 删除旧表
DROP TABLE IF EXISTS schedule_entries;

-- 3. time_assignments 已存在，确保 schema 正确
-- (AutoMigrate 会在启动时处理)

-- 4. 创建新 schedule_entries
CREATE TABLE schedule_entries (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at DATETIME,
    updated_at DATETIME,
    deleted_at DATETIME,
    semester_id INTEGER NOT NULL,
    schedule_version_id INTEGER NOT NULL,
    time_assignment_id INTEGER NOT NULL UNIQUE,
    classroom_id INTEGER NOT NULL,
    FOREIGN KEY (semester_id) REFERENCES semesters(id),
    FOREIGN KEY (schedule_version_id) REFERENCES schedule_versions(id),
    FOREIGN KEY (time_assignment_id) REFERENCES time_assignments(id),
    FOREIGN KEY (classroom_id) REFERENCES classrooms(id)
);
CREATE INDEX idx_se_version ON schedule_entries(schedule_version_id);
CREATE INDEX idx_se_classroom ON schedule_entries(classroom_id);

-- 5. 标记 migration 完成
INSERT OR REPLACE INTO schema_migrations (version, applied_at)
VALUES ('v0.6.0', datetime('now'));

COMMIT;
```

### 6.3 Snapshot 兼容

**旧 snapshot** (`schema_version = 'v0.5.4'` 或 `'v0.5.5'`):
- `entries_json` 包含旧 `[]ScheduleEntry` JSON
- v0.6.0 解析时检测 `schema_version`
- 旧版本: UI 标记 "Legacy"，禁止恢复（INV-MI2）
- 旧版本: 不尝试反序列化为新模型

**新 snapshot** (`schema_version = 'v0.6.0'`):
- `entries_json` 包含 `ScheduleSnapshotDTO` JSON
- `ScheduleSnapshotDTO.assignments[]` 中: FULL 有 `classroomId/classroomName`，TIME_ONLY 无（`omitempty + pointer`）

### 6.4 MoveService 适配

`MoveService` 当前直接读写 `ScheduleEntry`。适配方式:

- **读取**: 通过 `GetEnrichedScheduleEntries()` 获取展平视图
- **移动时间**: 更新 `TimeAssignment.DayOfWeek/StartPeriod/Span`
- **移动教室**: 更新 `ScheduleEntry.ClassroomID`
- **冲突检查**: 从 EnrichedScheduleEntry 中提取 time/room 占用
- **评分**: 通过 `EnrichedScheduleEntry` → `ScheduleEntry`（评分接口不变）

**适配复杂度分析** (来自 agent 深度分析):

MoveService 是受模型拆分影响最大的服务:
- `CheckMoveRequest` 仍然接受 `NewDay/NewPeriod/NewSpan/NewClassroom` 四个字段，签名不变
- `CheckMove` 内部分叉: 时间冲突检查走 TimeAssignment 表，教室冲突检查走 ScheduleEntry 表
- `MoveEntry` 分两次更新: `UPDATE time_assignments SET day=?, period=?, span=?` + `UPDATE schedule_entries SET classroom_id=?`
- `computeScoreDB` 改为 JOIN 查询: `SELECT ... FROM schedule_entries se JOIN time_assignments ta ON se.time_assignment_id = ta.id WHERE se.semester_id = ?`
- 性能影响: 当前单表全量扫描 → JOIN 两表全量扫描 (索引覆盖，可接受)

### 6.5 教室冲突唯一约束处理

**旧约束** (在 `schedule_entries` 上):
```sql
UNIQUE INDEX idx_schedule_room (classroom_id, semester_id, day_of_week, start_period)
```

**新模型下问题**: DayOfWeek 和 StartPeriod 移至 `time_assignments` 表，无法在单表上建立此约束。

**方案**: 应用层唯一性检查。在 `SchedulingService` 持久化路径中:
```go
// FULL 模式: 写入 ScheduleEntry 前检查教室+时间冲突
for _, alloc := range allocations {
    ta := taByLocalRef[alloc.LocalRef]
    // 检查: 同一教室(semester_id, classroom_id)在同一时间
    //       (day_of_week, start_period) 是否已被占用
    if existing := findConflict(tx, classroomID, dayOfWeek, startPeriod); existing != nil {
        return ErrRoomConflict
    }
}
```

- Gap: 当前有 DB 层唯一约束兜底，迁移后变为应用层检查（Go 侧 + 事务边界内）
- 风险评估: 🟡 中等。应用层检查在事务内，SQLite 串行写，不会出现 race condition。但不如 DB 约束简洁。

## 10. Open Questions

1. **EnrichedScheduleEntry 性能**: JOIN 3 表 (time_assignments + schedule_entries + teaching_tasks) 相比当前单表查询的额外开销。需要在 v0.5.7 数据量级下 benchmark。

## Appendix C: SA Solver Room Dependency Detail

Agent 深度分析确认: SA solver 的 2197 行代码中，教室耦合集中在每个 operator 的教室选择回路中。

**7 个评分维度中仅 1 个依赖教室**:
| # | 维度 | 依赖教室? | 说明 |
|---|------|----------|------|
| 1 | teacherPreferences | ❌ | 仅读 StartPeriod (早晚课) |
| 2 | courseSpacing | ❌ | 仅读 DayOfWeek (跨天分布) |
| 3 | teacherDays | ❌ | 仅读 DayOfWeek (天数) |
| 4 | lowFloorPref | ✅ | 读 ClassroomID → room.Floor |
| 5 | weekendAvoid | ❌ | 仅读 DayOfWeek |
| 6 | pePeriodPref | ❌ | 仅读 StartPeriod |
| 7 | studentFatigue | ❌ | 仅读 StartPeriod/Span/DayOfWeek |

**提取纯 TimeScheduler 需移除的教室回路** (v0.6.1):
- `sa_initial.go`: 移除 `for _, room := range rooms` 循环 + `Match()` + `canRoomFitCapacity()`
- `sa_neighbors.go`: 移除 `tryMove` 中的教室重选循环
- `sa_postopt.go`: 移除穷举搜索中的教室维度
- `sa_scorecache.go`: 移除 `classroomByID`/`maxFloor` + `low_floor_preference` 类别
- `sa_solver.go`: 移除 `roomOcc` map + `addOccupancy/removeOccupancy` 中的教室分支

**当前 `timeOnly` flag 的作用**: 仅跳过 `Match()` 调用（类型匹配），但仍遍历教室、选教室、分配 ClassroomID、追踪 roomOcc。v0.6.0 Adapter 保留此行为，v0.6.1 彻底移除。
