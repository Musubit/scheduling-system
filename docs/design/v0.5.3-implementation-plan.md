# v0.5.3 实施计划 (Implementation Plan)

> **创建**: 2026-07-13
> **关联设计**: [v0.5.3 URMF 设计文档](../design/v0.5.3-resource-matching-framework.md)

## P1: 数据模型

### 目标
追加数据库字段，不改变任何现有行为。

### 修改文件

| 文件 | 改动 |
|------|------|
| `backend/models/course.go` | +1 字段: `Category string` |
| `backend/models/classroom.go` | +1 字段: `Equipment string` |
| `backend/models/teaching_task.go` | +2 字段: `RequiredRoomType string`, `RequiredEquipment string` |

### 实施步骤

1. 在三个 model 文件中追加字段（全部 `default:''` / `type:text`）
2. 运行 `go build ./...` 确认编译通过
3. 启动应用，确认 GORM AutoMigrate 正常添加列
4. 验证旧数据读取正常（新字段为空值）

### 验收标准

- [ ] `go build ./...` 通过
- [ ] 现有 14 个测试全绿（无行为变更）
- [ ] 数据库新增 4 列，旧数据值为空
- [ ] 应用启动无报错

### 停止点
数据模型就绪，可以开始写 Matcher。

## P3: SA 接入

### 目标
SA 求解器 5 处 if-else 替换为 ResourceMatcher 调用。

### 修改文件

| 文件 | 改动 |
|------|------|
| `backend/services/sa_solver.go` | 删除 `getRequiredRoomType()`；共享场地判断改用 `IsSharedVenue()` |
| `backend/services/sa_initial.go` | 删除 L36-43 if-else，改用 `Match()` + `InferRoomType()` |
| `backend/services/sa_neighbors.go` | 删除 L139-144 if-else，改用 `Match()` |
| `backend/services/sa_postopt.go` | 删除 L309-319 + `roomTypeForCourse`，改用 `Match()` + `InferRoomType()` |
| `backend/services/scheduling_service.go` | 删除 L506-513 if-else，改用 `InferRoomType()` |

### 实施步骤

1. `sa_solver.go` — 删除 `getRequiredRoomType`，替换所有调用为 `InferRoomType`
2. `sa_solver.go` — `room.Type == "体育馆"` 判断改为 `IsSharedVenue(room)`
3. `sa_initial.go` — 替换 if-else 为 `Match()` 调用
4. `sa_neighbors.go` — 同上
5. `sa_postopt.go` — 同上 + 删除 `roomTypeForCourse`
6. `scheduling_service.go` — 替换 if-else 为 `InferRoomType()`
7. 运行 `go build ./...`
8. 运行全量测试

### 验收标准

- [ ] `go build ./...` 通过
- [ ] 现有 14 个回归测试全绿
- [ ] SA 对 seed 数据产出评分与 v0.5.2 基线一致（±0.01）
- [ ] `sa_delta_parity_test.go` — delta cache parity 通过
- [ ] `sa_postopt_test.go` — 后优化单调递增
- [ ] `sa_solver_test.go` — 88 sessions 全放置

### 停止点
SA 全部走 Matcher 路径，回归通过。

## P4: OR-Tools 接入

### 目标
Go 侧预计算 AllowedRooms，Python 删除匹配逻辑。

### 修改文件

| 文件 | 改动 |
|------|------|
| `backend/services/ortools_client.go` | ORToolsTask 加 `AllowedRoomIDs []uint`；ORToolsRoom 加 `Equipment string` |
| `backend/services/scheduling_service.go` | 构建 payload 时调用 `AllowedRooms()` 预计算 |
| `scheduler/solver.py` | 删除 `match_room()` + `SPECIALTY_ROOM_TYPES`，改读 `allowedRoomIds` |

### 实施步骤

1. `ortools_client.go` — ORToolsTask 加 `AllowedRoomIDs`；ORToolsRoom 加 `Equipment`
2. `scheduling_service.go` — 构建 payload 时对每个 task 调用 `AllowedRooms()`
3. `solver.py` — 删除 `match_room()` 函数和 `SPECIALTY_ROOM_TYPES` 常量
4. `solver.py` — HC7 改为读 `allowedRoomIds` 做 ID 过滤
5. 启动 OR-Tools 服务，运行 seed 数据排课
6. 验证 `allowedRoomIds` 长度与旧逻辑候选集一致

### 验收标准

- [ ] OR-Tools solver 对 seed 数据产出可行解
- [ ] 每个 task 的 `allowedRoomIds` 与 Go 侧 `AllowedRooms()` 结果一致
- [ ] Python 侧不再包含任何教室类型匹配逻辑
- [ ] `solver.py` 中不出现 "体育馆" / "实验室" / "机房" 硬编码字符串

### 停止点
OR-Tools 完全依赖 Go 预计算，一致性验证通过。

## 风险与回滚

| 风险 | 回滚方式 |
|------|----------|
| SA 评分波动 > ±0.5 | git revert P3 commit，Matcher 核心保留但 SA 不接入 |
| OR-Tools 不可用 | P4 可独立回滚，Python 恢复 match_room() |
| MoveService 误拒 | P5 可独立回滚，CheckMove 去掉 Match() 调用 |

每个 Phase 是独立 commit，可以单独 revert。
