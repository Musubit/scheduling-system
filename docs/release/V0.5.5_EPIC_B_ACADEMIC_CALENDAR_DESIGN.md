## 一、当前架构全景分析

### 1.1 现有 Semester 模型

**位置**：`backend/models/setting.go:6-11`

```go
type Semester struct {
    gorm.Model
    Name      string `gorm:"uniqueIndex;size:100;not null"`      // "2025-2026 第二学期"
    IsActive  bool   `gorm:"default:false"`                      // 布尔开关
    StartDate string `gorm:"size:20"`                            // "2025-09-01"（字符串！）
}
```

**3+2 技术债**：

| 问题 | 严重等级 | 说明 |
|---|---|---|
| `Name` 作为显示名 + 查找键 | 🔴 高 | `version_service.go` 以 `Where("name = ?", name)` 做文本匹配；名称带空格（"2025-2026 第二学期"）不一致，Seed 中 ScheduleEntry 引用的 Semester string 也与 `Name` 值完全一致，改名即失联 |
| `IsActive` 布尔开关 | 🔴 高 | 不能同时表达"当前学期"和"默认学期"；无生产数据时可派生但当前硬查；Seed 中 `IsActive: true` 固定第二学期 |
| `StartDate` 存为字符串 | 🟡 中 | `"2025-09-01"` 非 `time.Time`，不能直接做日期运算；WeekView 被迫用"当年 1 月 1 日"伪计算替代 |
| 无 AcademicYear / Term 字段 | 🟡 中 | 无法按学年筛选；学期排序依赖 `Name` 字符串而非结构化字段 |
| 无 TeachingWeeks 字段 | 🟢 低 | 周次上限硬编码在 Vue store（`schedule.ts:8` 中 `currentWeek < 20`）而非数据定义 |

### 1.2 Semester 依赖关系图

```
┌─────────────────────────────────────────────────────────────────────┐
│                            models.Semester                           │
│  Name / IsActive / StartDate(string)                                │
└──────┬──────────┬────────────┬──────────┬───────────┬──────────────┘
       │          │            │          │           │
       ▼          ▼            ▼          ▼           ▼
  teaching_task  schedule    snapshot   version    move_service
  .SemesterID  _entry      .Semester  _service   .computeScore
  (FK: uint)   .Semester   (string)   (string)    (string)
               (string)
```

### 1.3 关键发现：两套 Semester 引用系统并存

| 引用方式 | 使用方 | 问题 |
|---|---|---|
| `SemesterID uint` (FK) | `TeachingTask`, `ScheduleVersion` | 健康，直接关联主键 |
| `Semester string` | `ScheduleEntry`, `ScheduleSnapshot` | 🔴 健康问题！用字符串 `"2025-2026 第二学期"` 关联，唯一索引依赖此字段，与 Seed 硬编码绑定 |

`SchedulingConfig` 内部同时持有 `Semester string` 和 `SemesterID uint`——双重表达。

## 二、问题列表

### 2.1 必须解决的问题（Blocking）

| # | 问题 | 影响 | 解决方案方向 |
|---|---|---|---|
| P1 | `ScheduleEntry.Semester` 存为 string | 唯一索引依赖字符串，不与 Semester 主键 FK | 迁移为 `SemesterID uint` FK |
| P2 | `ScheduleSnapshot.Semester` 存为 string | 快照按学期筛选需字符串匹配 | 迁移为 `SemesterID uint` FK |
| P3 | `Semester.Name` 作为查找键 | 改名即断裂，version_service 依赖 name 做 match | 不再依赖 Name 查找，用 ID 或 AcademicYear+Term |
| P4 | 无 AcademicTerm 概念 | Solver 无法判断当前周是否属于"考试段"或"实践段" | 新建 `academic_term` 表 |

### 2.2 必须解决的派生问题

| # | 问题 | 影响 | 解决方案方向 |
|---|---|---|---|
| P5 | 前端 WeekView 用"当年 1 月 1 日 + week*7"计算日期 | 完全错误。若 Semester 从 9 月起，全年范围正确偏移 | 提供 `GET /semesters/:id/weeks` 端点，前端零计算 |
| P6 | 前端 `currentWeek = ref(8)` 硬编码 | 初始值 8 和上限 20 都是乱猜 | 后端提供 `CurrentWeek(sem, now)` |
| P7 | `SchedulingConfig.Semester` 与 `SemesterID` 双重表达 | 易出现不一致 | 统一为 SemesterID |

### 2.3 建议解决的问题（推荐但可延后）

| # | 问题 | 说明 |
|---|---|---|
| P8 | Seed 中 ScheduleEntry 的 Semester string 硬编码 | 重构后需统一改为 SemesterID |
| P9 | `StartDate string` → `time.Time` | 配合 Week/Date 派生公式使用 |

## 三、目标模型设计

### 3.1 持久化层（SQLite 表）

```
┌──────────────────────┐
│      Semester        │  ← 表名沿用 semesters（避免 migration 重命名）
├──────────────────────┤
│ ID          uint      │
│ AcademicYear string   │  ← 新："2025-2026"，格式^\d{4}-\d{4}$
│ Term        string    │  ← 新：FIRST / SECOND
│ StartDate   time.Time │  ← 改：string→time.Time，强制周一
│ TeachingWeeks int     │  ← 新：21（跨表锚点）
└──────┬───────────────┘
       │ 1:N
       ▼
┌──────────────────────┐
│   AcademicTerm       │  ← 新表
├──────────────────────┤
│ ID          uint      │
│ SemesterID  uint      │  ← FK → semesters.id
│ Season      string    │  ← AUTUMN / WINTER / SPRING / SUMMER
│ TermType    string    │  ← TEACHING / EXAM / PRACTICE
│ StartWeek   int       │  ← Semester 内连续编号
│ EndWeek     int       │
│ UNIQUE (SemesterID, Season, TermType)
└──────────────────────┘
```

**关键决策**（基于前期 grilling 结果）：

| 决策 | 内容 |
|---|---|
| AcademicTerm 日期不落库 | `StartDate/EndDate` 由 `Semester.StartDate + WeekNumber` 派生 |
| 能力字段不落库 | `EnableCourseScheduling()`、`IsExamPeriod()` 为 Go 方法（根据 TermType 派生） |
| AcademicYear 不独立建表 | `Semester.AcademicYear` 字符串，Validator 校验 `^\d{4}-\d{4}$` |
| TeachingWeek 不建表 | 作为 ViewModel 由 `services/academic_calendar/` 派生 |

### 3.2 派生层（不落库）

```go
// TeachingWeekView — 由 services/academic_calendar 生成，非持久化
type TeachingWeekView struct {
    WeekNumber int       // 1..TeachingWeeks
    StartDate  time.Time // 周一
    EndDate    time.Time // 周日
    TermID     uint      // 所属 AcademicTerm.ID
}
```

### 3.3 Go 方法层

```go
// —──── services/academic_calendar/ ─────

// DateOfWeek 返回第 week 周的周一日期
func DateOfWeek(sem models.Semester, week int) time.Time

// WeeksOfSemester 返回学期所有周的 ViewModel 列表
func WeeksOfSemester(sem models.Semester, terms []models.AcademicTerm) []TeachingWeekView

// CurrentSemester 判断当前时间所属学期
func CurrentSemester(semesters []models.Semester, now time.Time) *models.Semester

// CurrentWeek 返回当前周次
func CurrentWeek(sem models.Semester, now time.Time) int

// ValidateAcademicTerm 校验 Week 区间不重叠且覆盖 1..TeachingWeeks
func ValidateAcademicTerm(sem models.Semester, terms []models.AcademicTerm) error
```

### 3.4 API Contract（Vue / Solver 消费）

```
GET /api/semesters
  → [{id, academicYear, term, startDate, teachingWeeks}]
  
GET /api/semesters/current
  → {id, ...} (派生)
  
GET /api/semesters/:id/weeks
  → [{weekNumber, startDate, endDate, termId, capabilities}]
  
GET /api/semesters/:id/current-week
  → {weekNumber, termId}
  
GET /api/academic-terms/:id
  → {season, termType, startWeek, endWeek, capabilities}
```

### 3.5 Python Solver 消费方式

Solver 当前只消费 `StartWeek/EndWeek`（int），不依赖日期。计划：
- **当前阶段**：solver 输入不变，继续传 StartWeek/EndWeek 数值
- **后续阶段**：新增 `AcademicTermContext` 数组，Solver 按 TermType 判断是否排课

## 四、迁移策略

### 4.1 操作顺序

```
Step 1: 新增 AcademicTerm 模型（新文件，无破坏）
Step 2: 重构 Semester 模型（破坏性变更）
Step 3: 迁移 ScheduleEntry.Semester string → SemesterID（数据重写入）
Step 4: 迁移 ScheduleSnapshot.Semester string → SemesterID（数据重写入）
Step 5: 新建 services/academic_calendar/ 包（新文件）
Step 6: 更新前端 semester store + WeekView（硬编码清除）
Step 7: 重写 seed.go Semester + ScheduleEntry seed（适应新模型）
Step 8: 重生成 bindings
```

### 4.2 数据迁移决策

由于**无生产数据**（前序 Epic 已确认）：

| 表 | 迁移方式 |
|---|---|
| `semesters` | DROP TABLE + 重建 + 新 seed（干净） |
| `academic_terms` | 新建 + seed 每学期写 3 条 |
| `schedule_entries` | DROP TABLE + 重建 + 更新 ScheduleEntry 写入路径 |
| `schedule_snapshots` | DROP TABLE + 重建 |

**注意**：`semesters` 表不能简单 DROP——`TeachingTask` 和 `ScheduleVersion` 中有 FK 约束。操作需在关闭 FK 校验（SQLite 默认不启用 `PRAGMA foreign_keys`，但 GORM 可能在连接时启用）或事务中完成。

**推荐策略**：在 `database.go` 的 `AutoMigrate` 阶段：
1. 重命名旧表 `semesters` → `semesters_v054_backup`
2. AutoMigrate 会重建新 `semesters` 表
3. 执行新 seed 灌入数据
4. 验证后手动 DROP `semesters_v054_backup`

## 五、风险分析

### 🔴 高风险

| 风险 | 概率 | 后果 | 缓解 |
|---|---|---|---|
| R1: `ScheduleEntry.Semester` string 唯一索引导致全表重建 | 高 | 旧索引与新列不兼容，SQLite AutoMigrate 不自动重建复合索引 | DROP + 重建 `schedule_entries` 表（无生产数据，可接受） |
| R2: Vue 多处隐式依赖 `s.name` 作为学期标签 | 中 | 若 Name 改为派生，下拉框、筛选器、API 调用会断裂 | 前端全仓搜索 `.name` 引用，逐一确认替换为 `academicYear + term` 派生 |
| R3: `version_service.go:resolveSemesterID` 用 `name` 文本匹配 | 中 | Name 删除或格式变化后直接报错 | 整函数删除，调用方改为传 SemesterID |

### 🟡 中风险

| 风险 | 说明 |
|---|---|
| R4: Seed 中 Semester 种子 IsActive 逻辑被硬编码 | 重构后 `IsActive` 不存在，需改为 `CurrentSemester(now)` 派生 |
| R5: Wails binding 类型变动导致前端编译不通过 | 一次性重生成 + 前端构建验证 |
| R6: `SchedulingConfig.Semester` string 字段被前端序列化传递 | 需追踪前端 `scheduling.ts` 是否依赖此字段 |

### 🟢 低风险

| 风险 | 说明 |
|---|---|
| R7: `SnapshotService` 中 "find semester_id by name" 子查询 | 已扫描到，整段替换为 ID 直查 |
