# Domain Docs

工程技能在探索代码库时应如何消费本仓库的领域文档。

## 探索前先读这些

- **`CONTEXT.md`**（仓库根目录）— 领域术语表
- **`docs/adr/`** — 读取与当前工作区域相关的架构决策记录

如果以上文件不存在，**静默跳过**。不要标记其缺失，也不要建议主动创建。`/domain-modeling` skill 会在术语或决策实际确定时惰性创建它们。

## 文件结构

单上下文仓库：

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-xxx.md
│   └── 0002-xxx.md
└── src/
```

## 使用术语表的词汇

当输出中出现领域概念时（issue 标题、重构提案、假设、测试名称），使用 `CONTEXT.md` 中定义的术语。不要漂移到术语表明确避免的同义词。

如果所需概念不在术语表中，这是一个信号 — 要么你在创造项目不使用的语言（重新考虑），要么确实存在缺口（标记给 `/domain-modeling`）。

## 标记 ADR 冲突

如果输出与现有 ADR 矛盾，显式指出而非静默覆盖：

> _与 ADR-0007（xxx）矛盾 — 但值得重新考虑，因为…_
