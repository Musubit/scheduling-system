# ADR-0001: 使用模拟退火（SA）作为排课求解器

## 状态

已采纳

## 上下文

排课是 NP-hard 约束满足+优化问题。需要为湖北工业大学（19个学院，数十门课，数十位教师，11间教室）生成高质量课表。

候选方案：
- **OR-Tools CP-SAT**：Google 的工业级约束规划求解器，能找到可证明的最优或近似最优解
- **模拟退火（SA）**：经典启发式算法，纯Go实现，零外部依赖

## 决策

**采用模拟退火，放弃 OR-Tools。**

## 理由

1. **体积**：OR-Tools 需要捆绑 `ortools.dll` + `protobuf.dll` + `abseil_dll.dll` 等依赖，体积暴增 150MB+。当前 Wails 桌面应用仅几MB，150MB 的增量对教务员用户不可接受。

2. **编译兼容性**：OR-Tools 预编译包为 Visual Studio (MSVC) 设计，而项目使用 MinGW GCC (Go CGO)。MSVC 和 MinGW 的 C++ ABI 不兼容，编译失败。

3. **Go 生态**：OR-Tools 的 Go 绑定是实验性的，依赖 Bazel 构建系统。在 Windows 上 Bazel 支持非常有限，无法作为标准 `go get` 依赖使用。

4. **问题规模适配**：19个学院、数十门课的规模对 SA 完全够用。SA 通过贪心初始解 + 邻域搜索（移动+交换）+ 温度冷却，在 10,000 次评估内即可找到高质量解。

5. **可维护性**：纯 Go 实现，零 CGO/C++ 依赖。任何人 clone 仓库即可编译运行，无需安装任何外部 SDK。

## 后果

- **正面**：应用体积保持轻量，跨平台兼容性无退化，未来维护成本低
- **负面**：无法证明解的最优性（SA 不保证全局最优），但实际使用中 SA 结果质量已充分满足需求
- **可逆性**：如未来 OR-Tools Go 生态成熟或需求规模显著增大，可替换为 OR-Tools（工作量约 2-3天）
