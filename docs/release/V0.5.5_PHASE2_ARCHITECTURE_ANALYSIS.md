## 二、Current Semester Domain Analysis

### 2.1 Semester 承载的四重角色

`backend/models/setting.go:11-18` 定义的 Semester 当前承载四种正交语义：

| 角色 | 字段 | 消费方 | 消费状态 |
|---|---|---|---|
| **身份（Identity）** | `AcademicYear` + `Term`（复合唯一） | Seed 查找、UI 显示、`DisplayName()` | ✅ 活跃 |
| **分区 FK（Partition Key）** | `ID` | `ScheduleEntry.SemesterID` / `TeachingTask.SemesterID` / `ScheduleSnapshot.SemesterID` / `ScheduleVersion.SemesterID` | ✅ 活跃（Phase 1 已 FK 化） |
| **日历元数据（Calendar Metadata）** | `StartDate`、`EndDate` | **无任何后端读取点** | ⚠️ 死字段（Phase 2 首次消费） |
| **生命周期状态（Lifecycle）** | `Status`（active/archived/planned） | Seed 初值、`GetActiveSemester()` | 🟡 弱活跃 |

### 2.2 关键判断：Semester **不承担**教学周期计算

事实依据（backend 全量扫描）：

- 全局 `Grep '\.StartDate|\.EndDate'` **无任何业务代码匹配**——只在 `seed.go:60-62` 三处写入，无消费者
- 教学周次**仅**由 `TeachingTask.StartWeek/EndWeek int` 与 `ScheduleEntry.Weeks string ("1-16")` 承载
- 唯一周次解析器：`backend/services/move_service.go:381-418` 的 `parseWeeksRange` / `weeksOverlap`，硬编码回退 `[1, 20]`
- OR-Tools 与 SA Solver 均只消费 `StartWeek/EndWeek int`，**不消费 Semester 日期**（`ortools_client.go:61-62` / `sa_initial.go:244` / `scheduling_service.go:611`）
- 结论：Semester 与"教学周期"的耦合度 = **近零**

### 2.3 设计意涵

Semester 保持"身份 + 分区 FK + 日历元数据 + 生命周期"的纯粹语义即可，**不应在 Semester 上挂载周次计算能力**。AcademicTerm 只承担"学期内时间段分类"这一新维度；周次↔日期派生逻辑归 `academic_calendar` 无状态函数包。

## 三、Problem Definition

### 3.1 Phase 2 要解决的核心问题

| # | 问题 | 影响 |
|---|---|---|
| Q1 | `Semester.StartDate/EndDate` 是死字段，无消费点 | Phase 1 落库但未接入领域逻辑 |
| Q2 | 前端 `WeekView.vue:85-95` 用"当年 1 月 1 日 + week × 7"计算日期 | 秋季学期从 9 月起，此算法**完全错误** |
| Q3 | 前端 `stores/schedule.ts:8` `currentWeek = ref(8)` 硬编码 | 与当前时间无关，与 Semester 无关，纯乱猜 |
| Q4 | 无"学期时段分类"概念 | 无法区分"考试周"与"教学周"，为 Phase 3+ Solver TermType 感知留白 |

### 3.2 缺失概念与派生逻辑

**缺失概念**：
- `AcademicTerm`（每学期 3 段：秋季常规教学 / 秋季考试 / 冬季实践；或春季常规教学 / 春季考试 / 夏季实践）

**缺失派生函数**（纯 Go、无状态、不依赖 DB）：
- `DateOfWeek(sem, week)` — 第 N 周周一日期
- `WeeksOfSemester(sem, terms)` — 学期完整周次视图列表
- `CurrentSemester(semesters, now)` — 依 now 归属活跃学期
- `CurrentWeek(sem, now)` — 当前周次

### 3.3 Phase 2 明确不解决

- ❌ Holiday（法定节假日）— Phase 3+ 或独立 Epic
- ❌ Solver TermType 感知（考试周禁排课）— Phase 3+
- ❌ 前端 `s.name`/`s.isActive` 残留清理 — Phase 3
- ❌ 前端 WeekView / schedule store 消费后端派生数据 — Phase 3
- ❌ 前端 `types/index.ts:58-63` 旧 Semester 类型 — Phase 3
- ❌ `SettingsPage.vue` 学期管理 UI 改造 — Phase 3

## 四、Candidate Architecture

### 4.1 方案对比

| 方案 | 结构 | 优点 | 缺点 | 违反约束 |
|---|---|---|---|---|
| **A. Semester 内扩展 JSON 列** | 在 Semester 上加 `SeasonSegments TEXT` 存 JSON 数组 | 单表、迁移零风险 | 表达力弱；SQLite 上无法索引；违反第一范式；破坏 Phase 1 的 Semester 稳态 | 🟡 破坏 Phase 1 稳态 |
| **B. 新增 AcademicTerm 表（1:N）** | Semester 1:N AcademicTerm；每学期 3 段；日期不落库派生 | 规范化；与 EPIC_B 设计一致；能力方法纯 Go 派生；无破坏 | 需新表 + AutoMigrate + Seed | ✅ 全满足 |
| **C. AcademicCalendar 聚合根 + TeachingWeek 表** | 新建 Calendar 表 + Term + Week 层次；周次也落库 | 表达力最强 | TeachingWeek 落库 20×N 行冗余；违反"不建 TeachingWeek 表"约束；违反"不过度设计" | 🔴 违反用户约束 |

### 4.2 推荐 — 方案 B

**Rationale**：
- 与 ROADMAP Phase 2 目标（"AcademicTerm 表 + academic_calendar 派生包，不建 TeachingWeek 表"）**完全一致**
- 与 EPIC_B 原设计的核心结构一致，且吸收其"日期不落库派生"决策
- 尊重 Phase 1 Semester 稳态（Semester 不动）
- 单机 SQLite 场景下写入量极小（学期数 × 3），不引发性能问题
- 派生逻辑放独立包，便于 Phase 3+ Solver TermType 感知复用

## 五、Recommended Design

### 5.1 分层职责边界

```
┌──────────────────────────────────────────────────────────────┐
│  Semester（Phase 1 稳态，Phase 2 不动）                        │
│    职责：学期身份 + 日历元数据（StartDate/EndDate）+ 分区 FK    │
│    不承担：周次计算、时段分类                                    │
└──────────┬───────────────────────────────────────────────────┘
           │ 1:N (SemesterID FK)
           ▼
┌──────────────────────────────────────────────────────────────┐
│  AcademicTerm（Phase 2 新增）                                  │
│    职责：学期内时间段分类（Season × TermType + Week 区间）      │
│    不承担：日期落库、能力常量、跨学期逻辑                        │
└──────────────────────────────────────────────────────────────┘

   ┌────────────────────────────────────────────────────────┐
   │  services/academic_calendar/（Phase 2 新增）             │
   │    职责：无状态派生函数集                                  │
   │    不承担：DB 访问、事务、事件发布                          │
   │    输入：models.Semester / []models.AcademicTerm / time  │
   │    输出：TeachingWeekView / bool / int / error           │
   └────────────────────────────────────────────────────────┘
```

### 5.2 决策清单（每条附 rationale）

| 决策 | Rationale |
|---|---|
| Semester 保持不变 | Phase 1 已稳定；改 Semester 会引发 FK/binding/前端类型级联，超出 Phase 2 范围 |
| AcademicTerm 日期不落库 | 起止日期完全可由 `Semester.StartDate + StartWeek/EndWeek` 派生，落库即引入不一致风险 |
| 能力方法（`IsExamPeriod()` 等）为纯 Go | 常量派生零成本；落库会引入"数据 vs 常量"漂移 |
| 派生函数不依赖 DB | 纯函数便于单测、组合、并发安全；DB 交互留给上层 Wails service |
| 每学期固定 3 段 | ROADMAP 明确目标；对应"常规教学 + 考试 + 实践"的中国高校学期结构 |
| 不加 `Semester.TeachingWeeks` 字段 | 可由 `max(AcademicTerm.EndWeek)` 派生，避免冗余（详见 R2） |

## 六、Data Model Proposal

### 6.1 AcademicTerm 结构字段

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `ID` | uint | PK (gorm.Model) | 主键 |
| `SemesterID` | uint | `not null; index; uniqueIndex:idx_term` + FK → semesters.id | 归属学期 |
| `Season` | string | `size:10; not null; uniqueIndex:idx_term` | 季节枚举 |
| `TermType` | string | `size:20; not null; uniqueIndex:idx_term` | 时段类型枚举 |
| `StartWeek` | int | `not null` | 学期内连续编号，含 |
| `EndWeek` | int | `not null` | 学期内连续编号，含；`EndWeek >= StartWeek`（应用层校验） |
| `CreatedAt/UpdatedAt/DeletedAt` | — | gorm.Model 提供 | — |

### 6.2 常量枚举

**Season**（对应季节）：
```
AUTUMN  // 秋
WINTER  // 冬
SPRING  // 春
SUMMER  // 夏
```

**TermType**（时段类型）：
```
TEACHING  // 常规教学
EXAM      // 考试周
PRACTICE  // 实践/实习/见习
```

**每学期 3 段的规范化组合**（First 学期）：
- (AUTUMN, TEACHING) → StartWeek 1..N₁
- (AUTUMN, EXAM) → StartWeek N₁+1..N₂
- (WINTER, PRACTICE) → StartWeek N₂+1..N₃

**每学期 3 段的规范化组合**（Second 学期）：
- (SPRING, TEACHING)
- (SPRING, EXAM)
- (SUMMER, PRACTICE)

### 6.3 GORM 索引与约束

| 约束 | 类型 | 位置 |
|---|---|---|
| `(SemesterID, Season, TermType)` | 复合唯一索引 | tag `uniqueIndex:idx_term` |
| `SemesterID` | 普通索引 | tag `index` |
| SemesterID → semesters.id | 外键 | tag `foreignKey:SemesterID` |
| Week 区间不重叠、覆盖 1..max | 应用层软性校验 | `ValidateAcademicTerm` 函数 |

### 6.4 关系图

```
Semester (1) ────< (N) AcademicTerm
   │                       │
   │                       ├─ Season: AUTUMN/WINTER/SPRING/SUMMER
   │                       ├─ TermType: TEACHING/EXAM/PRACTICE
   │                       └─ [StartWeek, EndWeek] (学期内编号)
   │
   ├─ AcademicYear + Term (身份)
   ├─ StartDate/EndDate (日历元数据，Phase 2 首次消费)
   └─ Status (生命周期)
```

### 6.5 TeachingWeekView（不落库 ViewModel）

```
TeachingWeekView {
    WeekNumber int       // 1..max(EndWeek)
    StartDate  time.Time // 该周周一
    EndDate    time.Time // 该周周日
    TermID     uint      // 所属 AcademicTerm.ID
    Season     string    // 派生自 AcademicTerm.Season
    TermType   string    // 派生自 AcademicTerm.TermType
}
```

## 九、Migration Strategy

### 9.1 变更类型

**纯增量、零破坏**：
- 新增表：`academic_terms`
- 零字段增删：所有既有表结构不变
- 零外部数据：`AcademicTerm.SemesterID` FK 无孤儿风险（Seed 保证 Semester 先于 Term 创建）

### 9.2 迁移操作

| 步骤 | 操作 | 风险 |
|---|---|---|
| AutoMigrate | GORM 自动 CREATE TABLE `academic_terms` | 无 |
| Seed 幂等 | `FirstOrCreate` by `(semester_id, season, term_type)` | 无（复合唯一键保证） |

### 9.3 无生产数据 / 无回填

- 本项目为单机应用，无部署态生产库
- 用户本地 SQLite 若已存在：AutoMigrate 只 CREATE 新表；Seed 在既有 Semester 上补齐 3 段
- Seed 幂等：重复运行不重复插入，不改动既有 AcademicTerm 数据

### 9.4 回滚

若 Phase 2 出现问题：
- 前向：GORM 无 auto-drop，`academic_terms` 表可通过手动 DROP 或忽略
- 因为无其他表依赖 `academic_terms`，回滚即为无操作

## 十、Risk Analysis

### R1 — AcademicTerm 覆盖率 Validator 强度

**风险描述**：Seed 或未来用户可能定义出 `AcademicTerm.StartWeek/EndWeek` 有 gap 或 overlap 的数据（例：段 A [1-8]、段 B [10-12]，第 9 周悬空）。

**推荐处置**：
- **软性警告**：`ValidateAcademicTerm` 返回详细 error，但**不阻塞 Seed / DB 写入**
- 单元测试覆盖 3 种坏数据：gap / overlap / 越界（超出学期总周数）
- Phase 2 不引入运行时强校验；留 Phase 3 UI 编辑时使用

**Rationale**：Phase 2 无 UI 编辑入口，运行时强校验会给未来 UI 添加约束。当前只需保证 Seed 数据正确、函数在坏数据下有明确行为。

### R2 — 是否新增 `Semester.TeachingWeeks int` 字段

**风险描述**：EPIC_B 原设计（v0.5.4）提议在 Semester 上加 `TeachingWeeks int`。若不加，`WeeksOfSemester` 需通过 `max(AcademicTerm.EndWeek)` 派生。

**推荐处置：不加**

**Rationale**：
- 违反 Phase 1 Semester 稳态原则（Phase 1 刚固化 Semester 结构，Phase 2 立刻改会引发 binding / 前端类型漂移）
- 冗余数据源：`Semester.TeachingWeeks` 与 `max(AcademicTerm.EndWeek)` 存在漂移风险
- 派生开销可忽略：每学期 3 条记录，遍历成本 O(3)
- 若 Phase 3 UI 需要"学期总周数"作为编辑锚点，届时可再评估

### R3 — Phase 3 前端消费迁移的兼容性

**风险描述**：本 Phase 若暴露 `AcademicCalendarService` 到 Wails（§7.4），会生成新 binding；Phase 3 前端消费时若 API 签名再变，破坏兼容。

**推荐处置**：
- Phase 2 **不暴露 Wails service**，只完成 model + 派生函数 + Seed + 单测
- Phase 3 前端消费启动前，再一次性设计并暴露 Wails service，避免中间态
- 若必须 Phase 2 暴露：签名冻结为 `[]TeachingWeekView` / `*Semester` / `int`，Phase 3 不改

**Rationale**：Wails binding 变更影响前端 build，Phase 2 保持"后端纯粹增量"最安全。

### R4 — `CurrentSemester(now)` now 落学期区间外的降级

**风险描述**：now 若位于两个学期之间（暑假、寒假），无匹配学期。

**推荐处置**：函数明确定义降级策略 —— 返回 `nil`（由调用方决定回退到 Status=active 的学期或最近学期），并被单元测试覆盖。
